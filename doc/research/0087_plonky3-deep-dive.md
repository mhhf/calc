# Plonky3 Deep Dive: Custom Circuit Viability

## What is Plonky3?

A Rust toolkit for implementing polynomial IOPs (PIOPs) — provides primitives (PCS, fields, hash functions, FRI) for building STARKs or PLONK-style systems. Version 0.4.2. Not a proving system itself but a composable layer. Primary consumers are zkVMs (SP1, OpenVM), but the AIR interface is general enough for custom circuits.

GitHub: https://github.com/Plonky3/Plonky3 — 758 stars, very active (multiple commits daily as of March 2026), 37 crates.

## 1. LogUp / Lookup Argument Support

**Yes — built-in, via the `p3-lookup` crate.**

The `lookup` crate implements LogUp (logarithmic derivative lookup arguments). It was refactored into a first-class API around January 2026 (PR #1239: "integrate Lookup logic into the Air trait").

The `Lookup<F>` struct is the user-facing interface:
- `kind`: `Local` (within one AIR) or `Global` (across multiple AIRs)
- `element_exprs`: symbolic expression tuples being looked up
- `multiplicities_exprs`: how often each tuple appears
- `columns`: auxiliary column indices for the running sum

Users implement the `LookupAir` trait on their AIR:
- `add_lookup_columns()` — allocate permutation columns
- `get_lookups()` — define all lookups via `SymbolicAirBuilder`

The `LogUpGadget` implements `LookupEvaluator` and `LookupGadget`. It generates the permutation trace using batch inversion + parallel prefix sum (~200 lines internally). Two challenges (alpha, beta) are sampled automatically.

Test examples: `RangeCheckAir` (single column range check) and `AddAir` (tuple lookup verifying addition triples). These are in `lookup/src/tests.rs`.

**SP1 and OpenVM do not use `p3-lookup`.** SP1 documents its "cross-table lookups" as "inspired by Valida" — it rolls its own interaction-based permutation argument. The `p3-lookup` crate's built-in LogUp is newer and may not have been available or stable when SP1 and OpenVM built their lookup layers.

## 2. Multi-Stage Witnesses (Prover Challenges Between Stages)

**Not supported in the current design.**

Both `uni-stark` and `batch-stark` are single-stage: the prover accepts one `RowMajorMatrix` trace, optionally a preprocessed trace, then runs in one monolithic proving pass. The challenge sequence is fixed: lookup challenges → alpha (constraint folding) → zeta (opening point). There are no intermediate witness stages where new trace columns are generated after receiving a verifier challenge.

`StarkInstance<'a, SC, A>` holds: `air`, `trace` (one matrix), `public_values`, `lookups`. No multi-stage fields.

The `batch-stark` prover batches multiple independent AIRs together, but each individual AIR is still single-stage.

**Consequence for LogUp:** LogUp's permutation trace (auxiliary columns) is generated by the prover internally during `prove_batch` — it's not a separate "stage 2" that users control. The framework handles this transparently.

For true multi-stage RAP (Randomized AIR with Preprocessing), users would need to extend the framework or use `batch-stark` with cross-AIR global lookups as a workaround.

## 3. Writing an AIR: Interface and Boilerplate

The `Air<AB: AirBuilder>` trait requires implementing a single method:

```rust
fn eval(&self, builder: &mut AB);
```

`BaseAir<F>` additionally requires:
- `width() -> usize` — number of trace columns

Optional overrides: `preprocessed_trace()`, `main_next_row_columns()`, `preprocessed_next_row_columns()`, `num_constraints()`, `max_constraint_degree()`, `num_public_values()`.

Constraint writing uses a builder pattern:

```rust
// Assert polynomial expression equals zero
builder.assert_zero(local[0] * local[1] - local[2]);

// Conditional constraints (gate by selector)
builder.when(local[is_first_step]).assert_eq(local[val], initial_value);

// Transition constraints (between rows)
builder.when_transition().assert_eq(next[acc], local[acc] + local[val]);

// Batch constraints over arrays
builder.when(condition).assert_zeros::<N, _>(array::from_fn(|i| expr_i));
```

Trace variables are accessed via `MainWindow` (two-row view: local/next) and `PreprocessedWindow`.

**Keccak AIR as a complexity benchmark:** ~180 lines for `eval()`, covering 24 rounds of Theta/Rho/Pi/Chi/Iota steps. This is a realistic upper bound for a non-trivial circuit.

For a simple custom circuit (e.g., a Fibonacci sequence, range check, or arithmetic circuit), the boilerplate is minimal — 20-50 lines of Rust.

**Proof setup boilerplate** (from `examples/src/proofs.rs`):
1. Choose hash for MMCS (Keccak or Poseidon2)
2. Wrap for extension field (`ExtensionMmcs`)
3. Configure FRI params (log_blowup, etc.)
4. Generate trace (`RowMajorMatrix`)
5. Instantiate PCS (`TwoAdicFriPcs` or `CirclePcs`)
6. Build challenger
7. Construct config struct
8. Call `prove()` / `verify()`

This is ~30-50 lines of setup. The config is strongly typed and generic over field choice.

## 4. Custom (Non-zkVM) Circuit Examples

The official examples in `examples/src/` are hash function AIRs: Blake3, Keccak, Poseidon2. These are non-trivial but specialized.

The `lookup/src/tests.rs` contains `RangeCheckAir` and `AddAir` — minimal custom AIRs that show the lookup integration pattern.

No community showcase of non-zkVM custom circuits was found in public GitHub search results. The ecosystem is predominantly zkVM-focused. The tooling works for custom circuits but there is no library of reusable circuit gadgets (unlike, say, Circom's ecosystem).

**CertiPlonk** (NethermindEth, 13 stars, Lean) is the most notable adjacent project — a formal verification framework for Plonky3 constraint systems. Not a circuit library but shows the constraint system is being taken seriously beyond zkVMs.

## 5. Solidity / On-Chain Verifier

**No.** There is no Solidity or EVM verifier in Plonky3.

- The README contains no mention of on-chain verification
- No GitHub issues found for "solidity", "evm", or "smart contract verifier"
- The BabyBear/Mersenne31/Goldilocks fields are not EVM-friendly (Groth16/KZG over BN254 is the standard path for on-chain)

The `bn254` crate exists in the workspace, suggesting BN254 field arithmetic support, but there is no Solidity verifier companion. If on-chain verification is needed, the standard approach is to wrap the Plonky3 STARK in a Groth16/KZG proof over BN254 using an external recursion layer (SP1 does this with its Groth16 wrapping step).

## 6. Supported Fields

| Field | Notes |
|---|---|
| BabyBear | 31-bit, primary field for SP1/OpenVM; AVX2/AVX-512/NEON SIMD |
| KoalaBear | 31-bit variant; SIMD optimized |
| Mersenne31 | 31-bit; Circle STARK support; AVX2/AVX-512/NEON SIMD |
| Goldilocks | 64-bit; ~128-bit extension |
| BN254 | For recursion / SNARK wrapping |

All 31-bit fields have ~128-bit extension fields for FRI security. The `TwoAdicFriPcs` works with BabyBear/KoalaBear/Goldilocks. `CirclePcs` is specific to Mersenne31 (circle group structure).

Field choice is threaded through `StarkGenericConfig` as a type parameter — fully generic, no runtime dispatch.

## 9. Constraint Degree Limit

**No hard limit — determined dynamically from the AIR.**

`BaseAir::max_constraint_degree()` returns an optional hint. If provided, it skips symbolic evaluation. If not (returns `None`), the framework computes it symbolically. The value must be >= actual degree or the proof is invalid.

The degree affects the quotient domain size: a degree-d constraint with trace length N produces quotient polynomials of degree ~d*(N-1). The FRI blowup factor must accommodate this. In practice, degree 3 is the sweet spot for STARK efficiency (the LogUp constraint degree is 3).

There is no enforced ceiling — if you write degree-10 constraints, the framework will compute a larger quotient domain and the prover will be slower.

## 10. How SP1 and OpenVM Use Plonky3

**SP1:** Uses Plonky3 for field arithmetic, FRI, hash functions, and the basic AIR interface. Rolls its own cross-table lookup system ("inspired by Valida") — a permutation-argument-based approach rather than LogUp. SP1's prover is multi-chip: each RISC-V operation type is a separate AIR, and the lookups stitch them together. SP1 adds a Groth16 wrapping step for on-chain verification.

**OpenVM:** "The STARK backend and circuit writing interfaces are built on top of Plonky3." Modular zkVM without a fixed CPU — custom instruction extensions are added as new AIR chips. Also uses its own lookup infrastructure rather than `p3-lookup`.

Neither uses `p3-lookup` — they predate it or built their own systems before it was stabilized. The `p3-lookup` crate appears to be a newer, cleaner upstream abstraction.

## Higher-Level Wrappers / DSLs (Community)

**powdr** (formerly powdr-legacy, ~500 stars): DSL + compiler targeting multiple backends including Plonky3. Separate from Plonky3 itself but the most developed "write constraints in a language, get a Plonky3 proof" tool.

**stwo** (Starkware, separate project): Circle STARK implementation over Mersenne31. Competes with Plonky3 for the same space. Has its own AIR DSL direction.

**CertiPlonk** (NethermindEth, 13 stars): Lean-based formal verification of Plonky3 constraint systems. Not a circuit library.

**zkALU** (0 stars): A Rust ALU circuit on Plonky3 — the only found example of a non-hash non-zkVM custom circuit.

No mature circuit gadget library exists for Plonky3 (no Circom-like ecosystem). Custom circuits require writing Rust directly against the `p3-air` trait.
